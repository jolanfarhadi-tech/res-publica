# Review Process

## Purpose
Defines how any Knowledge Product, decision, or proposed change is reviewed before being treated as final.

## Background
Reused from the organization's existing Verification Model and Responsibility Evidence verification workflow — not a new review mechanism.

## Core Principles
No self-review (a reviewer is never the item's own author); evidence-based judgment; human sign-off mandatory.

## Definitions
Mirrors the Responsibility Evidence verification workflow (`brain/GOVERNANCE/RESPONSIBILITY_EVIDENCE_MODEL.md` §5): Created → Submitted → Human Verification → Accepted/Rejected → logged.

## Framework
Applies uniformly across Knowledge Products, RPCS certifications, and governance decisions alike.

## Workflow
1. An item is submitted for review.
2. A named reviewer, distinct from the author, assesses it against applicable standards (Ethics Charter, Data Policy, Values).
3. Accepted or rejected, with the outcome logged.

## Roles
**Reviewer** — conducts review. **Author/Contributor** — submits; never self-reviews.

## Inputs
Any item requiring review.

## Outputs
An accepted or rejected determination, logged.

## Governance
This is the standing mechanism referenced by every other document's own "Governance" section that requires "human sign-off" or "review."

## AI Integration
AI may assist review (e.g., citation-checking, consistency-checking) but does not itself constitute the required human review.
