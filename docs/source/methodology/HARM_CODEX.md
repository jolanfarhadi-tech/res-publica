# Harm Codex

## Purpose
The organization's standing taxonomy of documented, recurring harm mechanisms — institutional memory that persists across individual cases.

## Background
Confirmed as a real system: website material states its public explanation page shows "what the Harm Codex is (explanation only, NO data)" — confirming the Codex itself holds sensitive case-linked material not intended for public display. RPCS names "Codex Research" as a specialization track. No full taxonomy structure document exists in the repository.

**Status: Version 1.0 reconstruction** for taxonomy structure and mechanism-mapping detail; the Codex's existence, purpose, and public/private boundary are confirmed.

## Core Principles
Institutional memory over one-off findings; mechanism-level pattern documentation, not individual blame.

## Definitions
A Harm Codex entry documents a recurring mechanism identified across multiple accounts — distinct from a single Responsibility Annex, which documents one specific pattern instance in depth.

## Framework
Fed by Mechanism Identification (HARM Lifecycle stage) and by recurring patterns surfaced across multiple Responsibility Annexes.

## Workflow
1. A recurring mechanism is identified across multiple accounts or Annexes.
2. Researchers document it as a Codex entry (taxonomy category, known instances, aggregation of evidence).
3. The entry supports future Mechanism Identification (pattern-matching new accounts against known mechanisms).
4. Public explanation of the Codex's existence and purpose is available; underlying case data is not public.

**Version 1.0 proposal — taxonomy structure, pending approval:** entries are classified into three proposed categories: **institutional mechanisms** (a specific institution's recurring practice), **systemic patterns** (a pattern recurring across multiple institutions), and **procedural failures** (a process design flaw, not tied to one institution). A proposed entry template: mechanism name, category, first-identified date, number of linked Annexes, aggregated evidence summary (never raw case data) — reviewed and verified before entry, per `../governance/REVIEW_PROCESS.md`.

**Formal name:** this three-category taxonomy is formally named the **National Harm Taxonomy** and serves as the classification layer for the Annex lifecycle — see `brain/FOUNDATION/03_ANNEX_BLOCKCHAIN_CIVIC_CONTRIBUTION_ARCHITECTURE.md` §12. This is a naming/integration formalization only; the three categories themselves are unchanged.

## Roles
**Researcher** — documents entries. **Expert** — validates taxonomy classification. **Reviewer** — verifies entries before they're relied upon.

## Inputs
Recurring patterns across Responsibility Annexes and Mechanism Identification findings.

## Outputs
Harm Codex entries (institutional memory); pattern-matching support for future intake.

## Governance
Public-facing material explains what the Codex is; it never exposes underlying case data publicly, consistent with data-protection and trauma-informed standards.

## AI Integration
AI may assist in pattern-matching a new account against existing Codex entries; it does not create or validate a Codex entry independently.

## Examples
Reserved — pending approved case material. The Codex's own design principle is explanation-only public exposure; no example entry is published here.
