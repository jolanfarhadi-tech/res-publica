# Architecture of the Digital Res Publica

> **Status: Founder Constitutional Decision — not a discovered artifact.**
> This document was not derived from prior repository evidence. It was
> authored and introduced by the founder, Jolan Farhadi, on 2026-07-07,
> as a deliberate framing decision for how this repository is to be
> understood. It is recorded here explicitly as such, per the standing
> rule established in this project's working history: nothing may ever
> be presented as discovered if it was intentionally introduced.

The development of Res Publica is not merely a software engineering project.

It is the gradual digitalization of organizational formation.

Every instruction, architectural decision, constitutional document, ADR, specification, implementation guideline, review, and governance rule represents an organizational distinction.

Together, these distinctions form the architecture of a digital institution.

The implementation process therefore consists of much more than writing code.

It consists of translating organizational reality into executable digital structures.

The repository contains:

- Constitutional documents
- Governance models
- Organizational rules
- Domain models
- Responsibility structures
- Architectural decisions
- Implementation instructions
- Development workflows
- Verification procedures
- Organizational memory

Each document defines distinctions that determine:

- what may happen,
- who may act,
- how legitimacy is established,
- how responsibility is transferred,
- how participation is organized,
- how institutional memory is preserved.

The source code is only the operational layer.

The real architecture exists in the organizational distinctions embodied by the documentation.

The repository therefore represents the architecture of a Digital Res Publica rather than merely the implementation of a software platform.

Its purpose is to transform a social organization into an executable institutional system.

Software becomes the medium through which organizational architecture is realized.

**The repository is not primarily a software repository. It is an Organizational Architecture Repository.**

---

## Note on current state (added for accuracy, not part of the founder's original formulation)

As of 2026-07-07, this framing describes an intended and partially realized condition, not a fully completed one. The organizational distinctions in `brain/GOVERNANCE/`, the ADRs, and the domain model are encoded as executable logic in `src/domain/` and `src/modules/` — tested and internally consistent — but that logic is not yet connected to any user-facing surface, API, or persistence layer. Five ownership questions (ADR-027 through ADR-031) remain deliberately unresolved. The medium is real; its realization is in progress.
